Construire une application web complète en 2026, c'est une expérience radicalement différente de ce qu'on connaissait il y a seulement deux ans. Les outils IA ont transformé le processus de développement, et il est désormais possible de livrer un produit fonctionnel en une fraction du temps habituel.

Dans ce guide, nous allons construire une application Next.js complète — de l'initialisation du projet au déploiement — en utilisant l'IA comme copilote principal.

## La stack technique

Voici la stack que nous allons utiliser. C'est notre stack standard chez Wizycode pour les projets clients :

- **Next.js 15** (App Router) — framework React full-stack
- **TypeScript** — typage statique pour un code robuste
- **Tailwind CSS** — styling utilitaire rapide et cohérent
- **PocketBase** — backend léger avec base de données, authentification et API en temps réel
- **Claude Code** — assistant IA pour le développement

Cette stack a été choisie pour son équilibre entre productivité, performance et maintenabilité. Next.js fournit le rendu serveur et le routing. TypeScript attrape les bugs avant l'exécution. Tailwind élimine le besoin de fichiers CSS séparés. PocketBase fournit un backend complet en un seul binaire.

### Configuration TypeScript stricte

Un point important : configurez TypeScript en mode strict dès le départ. C'est plus exigeant, mais ça vous évitera des bugs insidieux plus tard.

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

L'IA respectera ces contraintes et produira du code correctement typé.

## Étape 2 : Architecture des composants

Avant de coder, il faut réfléchir à l'architecture. C'est ici que l'IA est particulièrement utile : elle peut vous aider à concevoir une architecture cohérente avant d'écrire la première ligne de code.

Demandez à Claude Code :

```
Propose une architecture de composants pour une app de gestion de tâches
avec : liste de projets, tableau kanban par projet, détail de tâche,
authentification utilisateur. Utilise les Server Components par défaut
et les Client Components uniquement quand c'est nécessaire.
```

### Server Components vs Client Components

La règle d'or avec Next.js App Router : **Server Components par défaut**.

Les Server Components s'exécutent côté serveur et n'ajoutent pas de JavaScript au bundle client. Ils sont parfaits pour :
- Afficher des données depuis la base de données
- Rendre du contenu statique ou semi-statique
- Les layouts et les pages de contenu

Les Client Components sont nécessaires uniquement pour :
- Les interactions utilisateur (clic, formulaire, drag & drop)
- Les hooks React (useState, useEffect)
- Les API du navigateur (localStorage, géolocalisation)

Claude Code comprend cette distinction et placera le `'use client'` uniquement là où c'est nécessaire.

## Étape 3 : Mise en place de PocketBase

PocketBase est un backend open-source qui tient dans un seul binaire. Il fournit :
- Une base de données SQLite
- Un système d'authentification complet
- Une API REST auto-générée
- Des événements temps réel via SSE
- Un tableau de bord d'administration

### Installation et lancement

```bash
# Télécharger PocketBase
wget https://github.com/pocketbase/pocketbase/releases/download/v0.25.0/pocketbase_0.25.0_linux_amd64.zip
unzip pocketbase_0.25.0_linux_amd64.zip

# Lancer le serveur
./pocketbase serve
```

PocketBase sera disponible sur `http://localhost:8090` avec un dashboard admin sur `http://localhost:8090/_/`.

### Créer le client TypeScript

Demandez à Claude Code de créer un client PocketBase typé :

```
Crée un client PocketBase typé pour Next.js avec :
- Connexion au serveur PocketBase
- Types pour les collections (projets, tâches, utilisateurs)
- Fonctions utilitaires pour les opérations CRUD
- Gestion des erreurs avec des types discriminés
```

Claude Code va produire quelque chose comme :

```typescript
import PocketBase from 'pocketbase';

const pb = new PocketBase(
  process.env.NEXT_PUBLIC_POCKETBASE_URL || 'http://localhost:8090'
);

export interface Project {
  id: string;
  name: string;
  description: string;
  created: string;
  updated: string;
  owner: string;
}

export interface Task {
  id: string;
  title: string;
  description: string;
  status: 'todo' | 'in_progress' | 'done';
  priority: 'low' | 'medium' | 'high';
  project: string;
  assignee: string;
  created: string;
  updated: string;
}

export type ApiResult<T> =
  | { success: true; data: T }
  | { success: false; error: string };

export async function getProjects(): Promise<ApiResult<Project[]>> {
  try {
    const records = await pb.collection('projects').getFullList<Project>();
    return { success: true, data: records };
  } catch (err) {
    const message = err instanceof Error ? err.message : 'Erreur inconnue';
    return { success: false, error: message };
  }
}
```

Ce pattern de résultat typé (`ApiResult<T>`) est crucial. Il force le code appelant à gérer explicitement les cas d'erreur au lieu de les ignorer silencieusement.

## Étape 4 : Pages et routing

Avec App Router, chaque dossier dans `app/` correspond à une route. Voici la structure typique :

```
app/
├── layout.tsx              # Layout racine
├── page.tsx                # Page d'accueil
├── (auth)/
│   ├── login/page.tsx      # Page de connexion
│   └── register/page.tsx   # Page d'inscription
├── dashboard/
│   ├── layout.tsx          # Layout dashboard
│   └── page.tsx            # Vue d'ensemble
├── projects/
│   ├── page.tsx            # Liste des projets
│   └── [id]/
│       ├── page.tsx        # Détail d'un projet (kanban)
│       └── tasks/
│           └── [taskId]/
│               └── page.tsx # Détail d'une tâche
```

Demandez à Claude Code de créer cette structure avec le routing complet :

```
Crée les pages suivantes avec App Router :
- Page d'accueil avec présentation du produit
- Pages auth (login/register) avec formulaires
- Dashboard avec liste des projets de l'utilisateur
- Page projet avec vue kanban (3 colonnes : à faire, en cours, terminé)
- Page détail tâche avec formulaire d'édition
Utilise PocketBase pour les données et l'authentification.
```

### Exemple de page avec données serveur

Une page Server Component qui charge des données depuis PocketBase :

```typescript
import { getProjects } from '@/lib/pocketbase';
import { ProjectCard } from '@/components/ProjectCard';

export default async function DashboardPage() {
  const result = await getProjects();

  if (!result.success) {
    return (
      <div className="p-8 text-center">
        <p className="text-red-600">
          Erreur lors du chargement des projets : {result.error}
        </p>
      </div>
    );
  }

  const projects = result.data;

  if (projects.length === 0) {
    return (
      <div className="p-8 text-center">
        <p className="text-gray-500">Aucun projet pour le moment.</p>
        <CreateProjectButton />
      </div>
    );
  }

  return (
    <div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6 p-8">
      {projects.map((project) => (
        <ProjectCard key={project.id} project={project} />
      ))}
    </div>
  );
}
```

Notez comment la gestion d'erreur est explicite et complète : cas d'erreur, cas vide, cas nominal. L'IA produit naturellement ce genre de code complet quand on lui demande.

### Server Actions pour les mutations

Pour les formulaires et les mutations de données, utilisez les Server Actions de Next.js :

```typescript
'use server';

import { revalidatePath } from 'next/cache';
import { createTask } from '@/lib/pocketbase';

export async function addTask(formData: FormData) {
  const title = formData.get('title') as string;
  const projectId = formData.get('projectId') as string;

  if (!title || !projectId) {
    return { error: 'Titre et projet requis' };
  }

  const result = await createTask({
    title,
    project: projectId,
    status: 'todo',
    priority: 'medium',
  });

  if (!result.success) {
    return { error: result.error };
  }

  revalidatePath(`/projects/${projectId}`);
  return { success: true };
}
```

## Étape 7 : Styling avec Tailwind

Tailwind CSS est l'allié parfait de l'IA. Sa syntaxe déclarative et utilitaire est parfaitement adaptée à la génération de code.

Quelques principes pour obtenir du bon CSS avec l'IA :

1. **Décrivez le résultat visuel** plutôt que les classes CSS : "carte avec ombre légère, coins arrondis, padding généreux"
2. **Référencez des designs existants** : "style similaire à la page de pricing de Linear"
3. **Spécifiez le responsive** : "3 colonnes sur desktop, 1 colonne sur mobile"

L'IA produit du Tailwind propre et cohérent. Elle utilise les bonnes classes utilitaires, gère le responsive avec les préfixes `sm:`, `md:`, `lg:`, et applique les bonnes pratiques de spacing et de typographie.

## Workflow quotidien avec l'IA

Voici le workflow que nous utilisons au quotidien chez Wizycode :

1. **Planification** : décrire la fonctionnalité à Claude Code et lui demander un plan d'implémentation
2. **Implémentation** : confier l'implémentation à Claude Code en mode agent
3. **Review** : relire le code généré, vérifier la logique et les edge cases
4. **Tests** : demander à l'IA d'écrire les tests, puis les vérifier
5. **Ajustements** : corriger les détails manuellement ou via l'IA
6. **Commit** : utiliser Claude Code pour créer un commit avec un message descriptif

Ce workflow nous permet de livrer des fonctionnalités 3 à 5 fois plus vite qu'en codant manuellement, tout en maintenant un niveau de qualité élevé.

## Pièges à éviter

### Ne pas relire le code généré

L'IA fait des erreurs. Pas souvent, mais ça arrive. Relisez toujours le code généré, surtout pour :
- La logique métier critique
- La gestion des permissions
- Les requêtes de base de données
- La validation des entrées utilisateur

### Tout déléguer sans comprendre

L'IA est un outil, pas un remplaçant. Vous devez comprendre ce que fait le code pour pouvoir le maintenir, le débugger et le faire évoluer. Si l'IA génère du code que vous ne comprenez pas, demandez-lui de l'expliquer avant de l'accepter.

### Ignorer les erreurs TypeScript

TypeScript est votre filet de sécurité. Si l'IA génère du code avec des erreurs TypeScript, ne les ignorez pas avec des `// @ts-ignore`. Demandez à l'IA de corriger le typage.

### Oublier les edge cases

L'IA gère bien les cas nominaux mais peut oublier les edge cases. Pensez toujours à :
- Que se passe-t-il si la liste est vide ?
- Que se passe-t-il si le serveur est indisponible ?
- Que se passe-t-il si l'utilisateur n'a pas les permissions ?
- Que se passe-t-il sur mobile avec un petit écran ?
