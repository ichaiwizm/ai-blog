## Définition

Un **skill Claude Code** est un bloc de compétences prêt à l’emploi pour une tâche précise (design frontend, refacto, doc, etc.).
Un skill peut embarquer :

* des commandes slash,
* des subagents spécialisés,
* des intégrations **MCP** (outils externes),
* des **hooks** qui réagissent à des évènements. ([GitHub][1])

L’idée : arrêter de refaire les mêmes prompts à la main et encapsuler une façon de travailler dans un module réutilisable.

---

## Anatomie d’un skill

### Commandes slash

Des raccourcis pour lancer des actions ciblées du skill.

```text
/ui-wireframe home-page ecommerce minimal
```

Exemples typiques dans un skill frontend :

* `/ui-wireframe` : propose une structure d’écran
* `/ui-refine` : raffine une maquette existante
* `/ui-audit` : audit de cohérence typographie/couleurs

---

### Subagents

Des “mini-Claude” spécialisés dans un sous-problème, par exemple dans un skill frontend :

* `design-ux` : se concentre sur le parcours utilisateur
* `design-ui` : se concentre sur la grille, la hiérarchie visuelle
* `design-accessibility` : vérifie contrastes, tailles de texte, ARIA

Tu peux les appeler directement dans Claude Code (suivant le skill) :

```text
@design-ui Crée une variation plus brutaliste de cette page de profil.
```

---

### MCP intégrés

**MCP (Model Context Protocol)** sert à connecter le skill à des outils externes : GitHub, DigitalOcean, une base de données, un moteur de recherche, etc. ([code.claude.com][2])

Un skill peut déclarer des MCP qu’il utilise en interne (par exemple : récupérer la liste des issues GitHub puis générer un dashboard d’admin).

---

### Hooks

Les **hooks** sont des scripts déclenchés automatiquement à des moments clés : début de session, après exécution d’un outil, fin de session, etc. ([code.claude.com][3])

Un skill peut les utiliser pour automatiser la routine :

* charger du contexte à chaque ouverture de projet,
* lancer des tests,
* générer un récap de session,
* mettre à jour la doc.

---

## Installer un skill

### 1. Ajouter (si besoin) un marketplace de plugins/skills

Un skill arrive souvent via un plugin dans un marketplace.

```text
/plugin marketplace add my-org/claude-plugins
```

### 2. Installer le skill

Selon la façon dont il est distribué, tu peux l’installer via la commande `/plugin` ou via le CLI :

```text
/plugin install frontend-design@my-org
```

ou côté terminal :

```bash
npx claude-plugins install @anthropics/claude-code-plugins/frontend-design
```

([claude.com][4])

### 3. Choisir le scope

Les skills suivent les mêmes scopes que le reste de la config :

| Scope   | Portée                    | Cas d’usage typique                  |
| ------- | ------------------------- | ------------------------------------ |
| Local   | Ce projet uniquement      | PoC, tests, env perso                |
| Project | Tous les users de ce repo | Standard d’équipe pour ce projet     |
| User    | Tous tes projets Claude   | Préférences “globals” sur ta machine |

## Skill **frontend-design**

### Objectif

Le skill **frontend-design** sert à casser le look “UI IA générique” (dégradés violets, gros cards ronds, Inter partout) et à produire des interfaces **nettes, variées et prêtes pour la prod**. ([claude.com][4])

Concrètement, il pousse Claude à prendre des décisions **fortes** en design :

* choisir une direction esthétique claire,
* utiliser une typo de caractère,
* construire une palette cohérente,
* proposer du HTML/CSS/JS ou des composants modernes, exploitables.

### Installation rapide

```bash
npx claude-plugins install @anthropics/claude-code-plugins/frontend-design
```

ou :

```text
/plugin install frontend-design@anthropics
```

Après ça, tu peux simplement dire à Claude :

```text
Utilise le skill frontend-design et propose 3 variantes de cette page dashboard.
```

---

### Tonalités de design disponibles (exemples)

Un même skill peut supporter plusieurs “modes” visuels, par exemple :

* **Brutally minimal** : ultra épuré, peu de couleurs, beaucoup d’espace blanc
* **Maximalist** : visuel dense, typographie expressive, grosses highlights
* **Retro-futuristic** : néons, formes géométriques, vibe années 80–90
* **Luxury / Refined** : palette sombre + accents dorés, serif élégantes
* **Playful / Toy-like** : couleurs vives, arrondis, micro-animations ludiques

Tu peux les appeler dans le prompt :

```text
Crée une landing page pour une app de budget perso, en mode “Luxury / Refined”.
```

---

### Exemples de prompts avec le skill frontend-design

```text
Utilise le skill frontend-design.
Crée un dashboard web pour une app de trading, mode “Brutally minimal”, prêt en React + Tailwind.
```

```text
Avec le skill frontend-design, design une page de settings avec dark mode,
réglages de notifications et gestion de comptes liés.
```

```text
Toujours avec frontend-design, propose une UI mobile first pour un
planner d’habitudes, style playful, avec un système de badges.
```

---

### Principes appliqués par le skill

Le skill encode plusieurs règles implicites :

* **Typographie** : pas de combo Arial/Inter par défaut, mais des pairs de typos avec une vraie personnalité.
* **Couleurs** : palette simple (2–3 couleurs dominantes), plus des accents ciblés, au lieu d’un arc-en-ciel informe.
* **Variables CSS / tokens** : couleurs, espacements, radius, z-index… centralisés dans des variables ou design tokens.
* **Anti-convergence** : bannir les patterns ultra récurrents dans les datas (dégradés violet/bleu, hero “SaaS générique”, etc.). ([Medium][5])

## Utiliser MCP avec un skill

### Rappel : c’est quoi MCP ?

**MCP** connecte Claude à des outils externes via un protocole standardisé : serveurs GitHub, recherche, cloud, etc. Le skill peut ensuite exploiter ces serveurs comme s’ils faisaient partie de son “cerveau étendu”. ([code.claude.com][2])

### Ajouter un serveur MCP

Exemple avec un serveur GitHub :

```bash
claude mcp add github --scope user
```

Exemple avec un serveur DigitalOcean (local au projet) :

```bash
claude mcp add digitalocean-mcp-local \
  -e DIGITALOCEAN_API_TOKEN=TON_TOKEN \
  -- npx "@digitalocean/mcp"
```

([DigitalOcean][6])

Tu peux voir ce qui est configuré :

```bash
claude mcp list
```

Et souvent déclencher l’UI d’authentification / gestion dans Claude Code en tapant :

```text
/mcp
```

### Exemple : skill frontend-design + MCP GitHub

Scénario :

1. Tu ajoutes un MCP GitHub.
2. Un skill “frontend-design” étendu sait appeler ce MCP pour :

   * lister les issues,
   * lire le contenu d’un fichier de config,
   * récupérer les labels existants.
3. Tu lui demandes :

```text
Utilise le skill frontend-design et le MCP GitHub.
Génère un dashboard de product management avec :
- colonne par label GitHub
- métriques avec le nombre d’issues par label
- un panneau latéral “Issue details”.
```

Le skill peut alors *réellement* se brancher sur ton repo, lire les données via MCP, et produire une UI adaptée à ton cas réel.

## Utiliser des hooks dans un skill

### Rappel : à quoi servent les hooks ?

Les **hooks** sont lancés automatiquement à certains moments, par exemple : ([code.claude.com][3])

* `SessionStart` : au lancement ou à la reprise d’une session Claude Code
* `PostToolUse` : après l’exécution d’un outil ou d’une commande
* `Stop` : à la fin d’une session

Ils sont déclarés dans la config Claude Code (fichiers de settings global, projet, local, etc.) et peuvent être packagés dans un skill.

### Exemple de hook “SessionStart” dans un skill frontend

Idée : à chaque fois que tu démarres Claude dans un projet frontend, le skill :

1. vérifie que les deps sont installées,
2. génère un fichier `design-context.md` avec :

   * la palette actuelle,
   * les typos utilisées,
   * les composants principaux.

Pseudo-config dans `.claude/settings.json` :

```json
{
  "hooks": {
    "SessionStart": [
      {
        "name": "prepare-frontend-design-context",
        "command": ["node", "scripts/build-design-context.js"],
        "when": { "projectType": "frontend" }
      }
    ]
  }
}
```

Le script Node peut lui-même appeler Claude via le skill pour synthétiser la doc.

### Exemple : hooks orientés “qualité”

Beaucoup de plugins/skills “sérieux” utilisent des hooks pour : ([Reddit][7])

* générer automatiquement un résumé de session (`Stop`),
* forcer un passage de linter ou de tests après certains outils (`PostToolUse`),
* tenir un `CHANGELOG.md` ou un `DESIGN_NOTES.md` à jour.

Tu peux très bien combiner ça avec un skill frontend :

* `PostToolUse` : après chaque grosse refonte d’UI, lancer un script qui :

  * prend le diff,
  * demande au skill d’expliquer ce qui a changé côté UX/UI,
  * met à jour un fichier `ui-changelog.md`.

## Créer ton propre skill

Structure typique :

```text
mon-skill-frontend/
├── skill.json          # manifest : nom, description, composants
├── commands/           # commandes slash exposées
├── agents/             # subagents spécialisés
├── mcp.json            # MCP servers à déclarer / configurer
└── hooks/              # scripts de hooks (sessionStart, stop, etc.)
```

([ClaudeCode Plugin Directory][8])

Tu décris dans `skill.json` :

* le contexte d’usage (ex : “design web B2B sobre”),
* les styles pris en charge,
* les MCP sur lesquels le skill s’appuie,
* les hooks qu’il veut enregistrer.

Ensuite tu publies ça via un marketplace, et les autres peuvent faire :

```text
/plugin marketplace add my-org/skills
/plugin install mon-skill-frontend@my-org
```

---

## Gestion des permissions

Dernier point important : **les skills héritent des permissions des plugins/MCP/hooks qu’ils utilisent**.

* Tu peux gérer ça avec une UX dédiée (`/permissions` suivant la config) pour voir quels skills ont accès à quoi.
* Pour les MCP, fais attention aux scopes (`local`, `project`, `user`) et aux variables d’environnement (tokens API, secrets).
* Un skill bien conçu doit être clair sur ce qu’il peut faire : “lit les issues GitHub”, “déploie sur DigitalOcean”, “écrit dans ce dossier seulement”, etc.

---

## En bref

* Les **skills** emballent une vraie façon de travailler (frontend, tests, docs…) dans un module installable.
* **MCP** donne aux skills des bras longs vers tes outils réels (GitHub, cloud, DB…).
* Les **hooks** automatisent la routine autour de ces skills (setup, qualité, récap de session).

Et le skill **frontend-design** est le bon point de départ si tu veux que Claude arrête de pondre des interfaces clonées sur tout le web et commence à produire du vrai design exploitable.
